## 第一部分 软件杯优秀项目书撰写思路

### 1. 撰写定位

软件杯类竞赛的优秀项目书，本质上不是营销宣传稿，而是兼顾评审阅读、答辩追问和工程可复现性的技术文档。文档应满足三个目标：

- 让评委在较短时间内理解项目解决了什么问题、为什么值得做。
- 让评委看到系统不是概念设计，而是已经形成较完整的软件工程实现。
- 让答辩环节中的架构、算法、模块、接口、测试和部署内容都能追溯到真实代码。

因此，项目书的主线应遵循“问题定义 - 需求分析 - 方案设计 - 系统实现 - 测试验证 - 创新特色 - 应用前景”的工程闭环，而不是单纯堆砌功能截图。

### 2. 目录规范总结

结合软件杯类赛事通知、历届参赛须知、优秀项目常见写法，以及通行的软件工程文档结构，项目书目录建议采用如下结构：

1. 项目概述
2. 项目背景与问题分析
3. 需求分析
4. 总体方案设计
5. 系统架构设计
6. 关键功能设计与实现
7. 数据设计
8. 核心技术与算法
9. 测试与验证
10. 创新点与特色
11. 应用价值与推广前景
12. 总结与展望

这样的目录有三个优点：

- 符合软件工程逻辑，评委阅读成本低。
- 便于把源码结构、接口设计、数据库设计、测试结果全部纳入同一主线。
- 答辩时可以直接按章节展开，不容易被问穿。

### 3. 优秀项目书的共性写法

优秀项目书通常有以下共性：

- 开篇先写清楚应用场景、目标用户和现实痛点，避免“先讲技术后讲问题”。
- 需求分析会同时覆盖功能需求、性能需求、安全需求和可扩展需求，而不只是罗列页面功能。
- 架构部分强调分层设计、模块边界、数据流和调用关系，体现工程化能力。
- 实现部分只重点写真正有技术含量和比赛辨识度的模块，不平均用力。
- 测试部分至少覆盖功能测试、接口测试、异常处理、稳定性或可用性验证。
- 创新点必须建立在真实实现上，不能把“计划做”写成“已经完成”。

### 4. 本项目的最佳写法策略

对于 AutoLearning V2，这个项目最适合的写法不是把它包装成“万能 AI 学习平台”，而是突出以下四条主线：

- 以学生个体差异为核心，构建画像驱动的个性化学习闭环。
- 以多智能体协同为核心，提升学习资源生成的并行效率和内容分工质量。
- 以知识图谱、错题复习、自适应难度和学情分析为核心，形成学习过程管理能力。
- 以讯飞数字人、语音、PPT、视频等能力接入为核心，增强学习交互和成果表达形式。

因此，本项目书的重点不应放在“页面好不好看”，而应放在：

- 多智能体编排机制如何工作。
- 学生画像如何驱动资源生成与答疑。
- 学习路径、题库、错题、复习、分析如何组成闭环。
- 讯飞生态能力如何嵌入学习场景。

### 5. 写作上的注意事项

- 不虚构数据，不写代码中不存在的功能。
- 对“已实现”“已接入”“具备接口封装但仍有演示降级逻辑”的内容要严格区分。
- 术语要统一，例如“学生画像”“学习路径”“Agent 编排”“SSE 流式输出”“知识图谱”“SM-2 复习算法”等不能混用。
- 关键指标尽量使用源码可验证事实，例如模块数量、路由数量、数据表数量、技术栈组成等。
- 创新点要写成“结构创新 + 机制创新 + 场景创新”，避免只写“用了大模型”。

## 第二部分 基于 AutoLearning V2 实际实现的项目书正文

### 1. 项目概述

#### 1.1 项目简介

AutoLearning V2 是一个面向高校学生与自主学习者的 AI 个性化学习平台。系统以学生画像为基础，以多智能体协同为核心，以知识图谱、题库训练、错题复习、学情分析和数字人交互为支撑，构建“诊断 - 生成 - 练习 - 反馈 - 调整”的学习闭环。

项目的核心目标不是简单回答问题，而是围绕学习全过程提供结构化支持，使系统能够根据不同学习者的知识水平、学习风格、薄弱环节和目标方向，自动生成适配的课程文档、思维导图、练习题、代码案例、学习路径和评估建议。

#### 1.2 项目定位

本项目定位为“AI Native 个性化学习平台”，其重点不在传统题库系统的静态内容管理，而在于：

- 通过学生画像实现内容适配。
- 通过多智能体架构实现多类型学习资源协同生成。
- 通过数据记录与分析形成持续优化的学习闭环。
- 通过讯飞数字人、语音和视频能力提升学习交互体验。

#### 1.3 项目目标

- 构建完整的个性化学习资源生成平台。
- 提供多智能体协同生成能力，支持流式、并行、可视化执行过程。
- 提供面向知识掌握与能力提升的路径规划、练习训练和错题复习功能。
- 接入讯飞生态能力，增强系统的人机交互形式和成果表达形式。

### 2. 项目背景与问题分析

#### 2.1 背景

当前高校学习平台和在线教育系统普遍存在三个问题：

- 学习内容供给静态化，难以匹配不同学生的知识基础和学习风格。
- 学习过程支撑碎片化，资料生成、练习训练、错题复盘、路径规划和学情反馈相互割裂。
- AI 工具通常只提供单轮问答，难以支撑结构化学习与持续跟踪。

随着大模型能力提升，AI 已具备较强的生成和辅导能力，但若没有清晰的任务拆分、上下文管理和学习过程建模，系统往往只能停留在“聊天机器人”阶段，无法形成稳定、可复用、可追踪的学习产品。

#### 2.2 现有方案不足

传统学习系统的问题主要体现在：

- 题库与讲义固定，更新成本高，个性化程度低。
- 学习路径依赖人工设计，无法动态适配不同学生。
- 错题本只能记录错误，不能自动安排复习节奏。
- 学情分析往往停留在简单统计，缺少对知识点、维度和阶段进展的综合分析。
- 语音、数字人和视频展示通常与学习业务脱节。

#### 2.3 本项目解决的问题

AutoLearning V2 主要解决以下问题：

- 如何根据学生画像动态生成更适配的学习内容。
- 如何将不同类型的学习资源生成任务拆分给专业 Agent 并高效协同。
- 如何把学习路径、题库训练、错题复盘、知识图谱和数据分析串联成闭环。
- 如何将讯飞数字人、语音合成、语音识别、PPT 生成和视频生成能力融入真实学习场景。

### 3. 需求分析

#### 3.1 目标用户

- 高校学生，尤其是需要自学编程、算法、机器学习等课程的学生。
- 具有明确学习目标但缺乏系统化学习路径的自主学习者。
- 希望进行个性化辅导、练习生成、学习分析和阶段总结的学习用户。
- 在教学、竞赛或培训场景中需要快速组织教学材料的师生群体。

#### 3.2 功能需求

根据项目实际实现，系统核心功能需求如下：

- 用户注册、登录、身份认证与角色区分。
- 学生画像创建、修改、查询与可视化展示。
- 智能对话与流式答疑。
- 多 Agent 协同生成课程文档、思维导图、练习题、代码案例等学习资源。
- 学习路径生成、阶段进度更新与路径查询。
- 题目生成、答案提交、错题查看、错题总结与邮件发送。
- 基于 SM-2 算法的复习调度与变式题生成。
- 轻量级知识图谱查询、学习顺序获取与路径辅助规划。
- 学情分析看板，包括答题统计、趋势分析、知识点掌握度和路径进度。
- 代码沙箱在线运行。
- 讯飞数字人、语音识别、语音合成、视频生成和 PPT 报告生成。

#### 3.3 非功能需求

- 系统应支持流式输出，提升资源生成和答疑的交互实时性。
- 系统应支持模块化扩展，便于新增 Agent、接口和前端页面。
- 系统应具备基础安全控制，包括 JWT 身份认证和密码哈希存储。
- 系统应支持本地部署与 Docker 部署。
- 系统应在外部大模型通道异常时具备基础降级能力。

#### 3.4 性能与可扩展需求

- 支持多 Agent 并行执行资源生成任务。
- 支持后续扩展更多学科知识图谱和资源类型。
- 支持接入不同大模型通道。
- 支持继续扩展数据库、缓存、知识库和推荐机制。

### 4. 总体方案设计

#### 4.1 方案总览

系统采用“前端交互层 - 后端接口层 - 智能编排层 - 业务服务层 - 数据存储层”的分层设计。

总体设计思想如下：

- 前端负责页面交互、流式内容展示、数据可视化和学习操作承载。
- FastAPI 后端负责接口组织、认证授权和业务协同。
- Orchestrator 编排器负责多 Agent 注册、调度与事件汇聚。
- 各类服务模块负责大模型调用、画像处理、知识检索、复习调度、分析聚合和讯飞能力集成。
- SQLite 负责核心业务数据持久化，轻量知识库采用内存检索结构。

#### 4.2 技术路线

- 前端：React 18、TypeScript、Vite、TailwindCSS、Recharts、Markmap、KityMinder。
- 后端：FastAPI、Uvicorn、SQLAlchemy、Pydantic。
- 认证：JWT、bcrypt。
- 大模型通道：DeepSeek 为主，讯飞星火为备。
- 语音与数字人：讯飞虚拟人、讯飞语音合成、讯飞语音识别、讯飞视频生成、讯飞 PPT。
- 数据存储：SQLite，Redis 为可选配置。

### 5. 系统架构设计

#### 5.1 整体架构

系统整体可概括为四层：

- 接入层：Web 前端页面、学习工作台、资源中心、知识图谱、数字人交互、学情分析和管理后台。
- 编排层：Orchestrator 与 7 个专业 Agent。
- 服务层：画像服务、LLM 客户端、RAG 检索、知识图谱、自适应难度、错题复习、学情分析、讯飞服务等。
- 数据层：SQLite 业务表、轻量知识库、静态文件与本地资源。

#### 5.2 前端架构

前端采用 React 单页应用架构，代码中共包含 17 个页面组件，覆盖：

- 首页与登录注册页面。
- 学生画像页面。
- 学习工作台页面。
- 学习路径页面。
- 题库页面与错题页面。
- 学习资源中心页面。
- 学情分析页面。
- 知识图谱页面。
- 讯飞对话页面与数字人页面。
- 学习视频页面。
- 管理员页面与管理题库页面。

前端重点特点包括：

- 支持流式消息消费与增量渲染。
- 支持 Markmap 与 KityMinder 进行知识结构可视化。
- 支持 Recharts 展示趋势、雷达图和统计图表。
- 对部分外部能力存在前端降级或演示逻辑，保证外部服务不可用时页面仍可运行。

#### 5.3 后端架构

后端采用 FastAPI 实现，当前代码中包含 23 个 API 模块、76 个路由端点。主要接口域包括：

- 认证与用户管理。
- 学生画像。
- 学习路径。
- 题库与错题。
- 学习资源生成。
- 多 Agent 流式协作。
- 智能对话。
- 学情分析。
- 知识库与知识图谱。
- 代码沙箱。
- 讯飞数字人、语音、视频和 PPT。
- 管理后台。

#### 5.4 多智能体架构

系统实现了 7 个专业 Agent：

- ProfileAgent：学生画像分析。
- DocAgent：课程文档生成。
- MindmapAgent：思维导图生成。
- QuestionAgent：练习题生成。
- CodeAgent：代码案例生成。
- PathAgent：学习路径规划。
- TutorAgent：智能答疑辅导。

所有 Agent 通过统一的 BaseAgent 抽象基类和 SharedContext 共享上下文工作，并由 Orchestrator 统一注册和调度。

#### 5.5 编排机制

Orchestrator 支持三种模式：

- 单 Agent 执行。
- 多 Agent 并行执行。
- 多 Agent 串行执行。

在“生成全部资源”的场景中，系统先运行 ProfileAgent 完成画像分析，再并行执行文档、导图、题目和代码四个 Agent，并通过队列汇合事件流，最终将结果汇总返回。

#### 5.6 事件与流式输出机制

系统采用 SSE 作为流式推送机制。核心事件包括：

- agent_start
- agent_thinking
- agent_chunk
- agent_complete
- agent_error
- orchestrator_plan
- orchestration_complete

该机制既提升了交互实时性，也便于前端可视化展示 Agent 的执行过程。

### 6. 核心功能设计与实现

#### 6.1 用户认证与角色管理

系统实现了注册、登录、退出和当前用户信息查询功能。认证使用 JWT 令牌，密码使用 bcrypt 哈希存储，角色分为 student 与 admin 两类。管理员接口采用额外的角色校验。

该模块为后续所有学习功能提供统一权限基础。

#### 6.2 学生画像模块

学生画像是本项目个性化能力的核心基础。画像数据包括：

- 基本信息：姓名、年龄、年级、学校、专业。
- 学习特征：学习风格、知识水平、学习目标。
- 扩展画像数据：兴趣点、薄弱点、优势点、维度数据等 JSON 结构。

系统支持画像创建、查询、修改和图表展示。更重要的是，画像不仅用于展示，还会被注入到 Agent 与聊天上下文中，直接影响资源生成策略。

#### 6.3 画像驱动的自动更新机制

在对话结束后，系统会调用画像服务，从对话文本中提取兴趣、薄弱点、优势项、知识水平和学习风格等信息，并增量更新到数据库中。

该机制使系统具备持续学习用户特征的能力，从而形成“交互越多，画像越准，推荐越个性化”的闭环。

#### 6.4 智能对话与学习辅导

系统提供普通对话接口和流式对话接口。TutorAgent 在响应时会自动读取学生画像，并按用户水平调整解释深度、术语密度和表达方式。

对话模块的价值不止在于回答问题，而在于：

- 将用户画像纳入回答过程。
- 保存会话历史。
- 支持对话后再分析和画像更新。
- 可以与语音、数字人等能力进一步结合。

#### 6.5 多 Agent 学习资源生成

学习资源生成是本项目最具辨识度的核心能力之一。系统已实现的主要资源类型包括：

- 课程文档。
- 思维导图。
- 练习题。
- 代码案例。
- 拓展阅读。
- 评估报告。

其中，多 Agent 流式协作接口主要覆盖四类并行生成资源：文档、导图、题目、代码案例；其他资源通过独立业务接口实现。

##### 6.5.1 文档生成

DocAgent 输出 Markdown 格式的结构化课程文档，通常包含：

- 课程概述。
- 学习目标。
- 核心知识点。
- 重点难点解析。
- 应用案例。
- 小结回顾。
- 延伸阅读。

##### 6.5.2 思维导图生成

MindmapAgent 输出兼容 Markmap 的 Markdown 结构，前端支持可视化渲染。系统中同时保留了 KityMinder 相关静态资源，体现出图谱化学习表达的扩展空间。

##### 6.5.3 练习题生成

QuestionAgent 能输出多题型结构化题目数据，包括：

- 选择题。
- 填空题。
- 简答题。

题目数据包含题干、选项、答案、解析、知识点和难度等字段，并存储到数据库中，便于后续提交、统计和错题复盘。

##### 6.5.4 代码案例生成

CodeAgent 负责输出可运行的代码案例。前端资源中心还集成了代码运行器，可直接调用代码沙箱在线运行 Python 代码，从而将“讲解 - 示例 - 运行验证”打通。

#### 6.6 学习路径规划

PathAgent 负责输出结构化学习路径，按阶段给出：

- 阶段名称。
- 阶段目标。
- 知识点列表。
- 学习活动。
- 预计时长。
- 难度。
- 前置依赖。
- 里程碑与评估方式。

后端支持学习路径生成、路径查询、阶段进度更新和用户路径列表查询，前端配套提供路径展示页面。

#### 6.7 知识图谱与学习顺序推荐

系统实现了轻量级知识图谱模块，当前内置了约 20 个机器学习相关知识节点，并通过有向无环图表示前置依赖关系。

知识图谱支持：

- 节点与边的查询。
- 前置知识获取。
- 基于拓扑排序的学习顺序生成。
- 相关知识点搜索。
- Mermaid 图代码导出。
- 统计信息输出。

这一模块为学习路径规划提供了结构化支撑，也增强了系统的知识组织能力。

#### 6.8 题库训练与答案提交

题库训练模块支持根据学习路径阶段动态生成题目，并记录用户作答情况。答案提交后会产生 AnswerRecord 记录，同时触发维度更新任务，用于后续画像修正和分析。

这使系统不再是单向生成内容，而是能采集学习反馈数据。

#### 6.9 错题本与错题总结

系统提供错题列表、错题复习分析、错题总结报告生成和邮箱发送等能力。

其中，错题总结功能会聚合用户历史错误记录，并通过大模型生成错题原因分析、知识点拆解、学习建议和阶段复习方案，从而提升错题本的复盘价值。

#### 6.10 基于 SM-2 的复习调度

系统在 ReviewSchedule 表的基础上实现了间隔重复复习机制，采用经典 SM-2 算法，根据回答质量、重复次数、难度因子和间隔天数动态更新下次复习时间。

该模块的意义在于把“做错了再看一遍”升级为“按记忆规律自动安排复习”，增强长期学习效果。

#### 6.11 自适应难度引擎

系统根据用户在特定知识点上的历史答题正确率、平均答题时间和近期样本数量，动态给出题目难度推荐。

该策略主要依据：

- 正确率高则建议提升难度。
- 正确率低则建议降低难度。
- 正确率中等则保持难度。
- 样本数决定推荐置信度。

这种机制有助于避免题目始终过易或过难，提高训练效率。

#### 6.12 学情分析看板

学情分析服务会聚合用户各类学习数据，当前可输出：

- 总答题数与正确率。
- 学习时长估算。
- 知识点掌握度。
- 最近 30 天学习趋势。
- 薄弱维度分析。
- 学习路径进度。
- 对话统计。

前端分析页面通过图表、热力图和雷达图等方式进行可视化展示，使学习反馈更加直观。

#### 6.13 代码沙箱

系统提供 Python 在线执行沙箱，支持：

- 基础安全关键字检查。
- 临时文件执行。
- 10 秒超时控制。
- 输出、错误和执行时间返回。

该能力与代码案例生成模块结合后，可以形成“AI 生成代码 - 用户在线运行 - 结果即时验证”的学习体验。

#### 6.14 轻量级 RAG 知识库

系统实现了轻量级 RAG 服务，不依赖外部向量数据库，而是使用基于字符与 n-gram 的相似度检索策略，在内存中完成文档分块、索引与查询。

其优点是：

- 部署轻量。
- 依赖少。
- 易于本地演示和教学实验。

其当前定位更适合小规模知识库与教学原型场景，为后续升级到向量数据库留下了空间。

#### 6.15 讯飞数字人与多模态能力接入

本项目深度接入讯飞生态，已封装的能力包括：

- 讯飞星火对话通道。
- 讯飞数字人会话启动、发言、保活和停止。
- 讯飞语音识别。
- 讯飞语音合成。
- 讯飞视频生成。
- 讯飞 PPT 大纲生成。

需要如实说明的是：

- 后端层面已具备真实接口封装和调用逻辑。
- 前端数字人页面中仍保留了演示视频和部分降级模拟逻辑，以保证外部服务不可用时的可展示性。

这种设计提高了演示稳定性，但也说明前端数字人交互距离完全生产级联调仍有继续完善空间。

### 7. 数据设计

#### 7.1 数据库概况

系统当前使用 SQLite 作为主存储，基于 SQLAlchemy ORM 进行建模。核心业务表共 8 张：

- users
- student_profiles
- learning_paths
- questions
- answer_records
- learning_resources
- conversations
- review_schedules

#### 7.2 主要数据实体说明

##### 7.2.1 用户表

用于存储用户名、密码哈希、角色、邮箱和时间戳信息，是认证和权限控制基础。

##### 7.2.2 学生画像表

用于存储学生静态信息和动态画像数据，是个性化推荐和资源生成的核心输入。

##### 7.2.3 学习路径表

用于存储学习路径标题、学科、难度、阶段信息、当前阶段和状态。

##### 7.2.4 题库与答题记录表

题库表存储题目结构化内容，答题记录表存储用户答案、正误、耗时和重试信息，是学情分析、错题本和自适应难度的重要数据来源。

##### 7.2.5 学习资源表

用于存储文档、导图、代码案例、评估报告等生成内容或其链接信息。

##### 7.2.6 对话记录表

用于记录对话内容、角色和会话信息，为学习辅导与画像更新提供依据。

##### 7.2.7 复习调度表

用于存储知识点复习计划、难度因子、间隔、复习次数、下次复习时间等信息，支撑 SM-2 算法实现。

### 8. 核心技术与关键算法

#### 8.1 SharedContext 共享上下文机制

SharedContext 用于在多 Agent 间共享如下数据：

- user_id
- topic
- student_profile
- knowledge_level
- learning_style
- results
- execution_log
- scratchpad

该设计保证了 Agent 之间无需直接相互调用，仍能共享必要信息，降低耦合度。

#### 8.2 LLM 双通道降级策略

统一 LLM 客户端采用：

- DeepSeek 主通道。
- 讯飞星火备通道。

当主通道调用异常时，系统自动切换到备通道，以提高可用性。该设计对上层 Agent 是透明的。

#### 8.3 拓扑排序学习顺序算法

知识图谱模块通过反向搜索前置知识点，并在子图上执行 Kahn 拓扑排序，从而生成从基础到目标的学习顺序。该算法适合处理前置依赖明确的知识体系。

#### 8.4 SM-2 间隔重复算法

系统根据回答质量、重复次数和难度因子动态计算下次复习间隔，用于提升长期记忆效果。这一机制把传统错题本升级为主动复习系统。

#### 8.5 自适应难度规则引擎

系统依据知识点维度的答题正确率、用时和样本数综合判断推荐难度。尽管当前实现仍以规则为主，但已经具备动态调节能力，适合作为后续智能推荐的基础。

#### 8.6 轻量级 RAG 检索

系统采用文本分块、关键词与 n-gram 相似度计算的方式，在不引入重型向量数据库的前提下完成小规模知识检索，降低了部署成本。

- 数字人用于实时互动与视频化表达。
